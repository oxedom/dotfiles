# Claude Code and Codex compatibility

The `skills/` and `claude-global/` trees are the tracked source ecosystem.
`init` exposes them under `~/.claude`; `agent-sync` then projects compatible
resources into Codex's native locations. Run `agent-sync` after adding,
renaming, or removing a command or subagent. Existing skill edits flow through
their symlinks immediately.

## What is shared

| Capability | Claude Code | Codex | Sync method |
|---|---|---|---|
| Skills | `skills/<name>/SKILL.md` via `~/.claude/skills` | `.agents/skills/<name>/SKILL.md` | Symlinked skill folder |
| Project instructions | `CLAUDE.md` | `AGENTS.md` | File symlink when no `AGENTS.md` exists |
| Legacy commands | `claude-global/commands/<name>.md` | `.agents/skills/<name>/SKILL.md` | Thin skill adapter that reads the source command |
| Custom subagents | `claude-global/agents/<name>.md` | `.codex/agents/<name>.toml` | Thin TOML adapter that reads the source agent |

Both products support the open Agent Skills format and both follow symlinked
skill folders. This is the high-fidelity part of the bridge.

## Gaps that cannot be solved with a symlink

- Claude subagents use Markdown plus YAML fields such as `tools`, Claude model
  aliases, `permissionMode`, and `color`. Codex agents use TOML with
  `name`, `description`, and `developer_instructions`. The adapter keeps the
  instructions live and maps clearly read-only agents to Codex's read-only
  sandbox, but Claude tool allowlists and model aliases do not transfer.
- Claude command interpolation such as ``!`git status``` and `$ARGUMENTS` is
  Claude-specific. The Codex wrapper instructs Codex to run dynamic commands
  itself and treat trailing prompt text as arguments.
- `settings.json` and `config.toml` are different schemas with different
  features. Model selection, permissions, trust, TUI preferences, voice,
  notifications, and status-line configuration stay product-specific.
- Claude hooks and Codex hooks have different lifecycle events and enforcement
  behavior. They should be ported and tested individually, not linked.
- Claude marketplaces/plugins and Codex plugins have separate installation and
  authorization state. Plugin-bundled skills are not automatically projected.
- Claude's general `@path` import syntax is not a portable `AGENTS.md` feature.
  The sync script recognizes top-level global imports that point directly to a
  `SKILL.md` and exposes those as Codex skills; other imports still need a
  product-specific wrapper. Shared project instructions should use plain
  Markdown where possible.
- Conversation history, memories, approvals, and remote/cloud state are
  product-specific and are intentionally not synchronized.

## Commands

```bash
# User-level catalog and ~/.claude resources
agent-sync
agent-sync --check

# One repository
agent-sync --project /path/to/repo
agent-sync --project /path/to/repo --check
```

The sync is conservative: it updates only files marked as generated by the
script, preserves hand-written destinations, and reports name/path conflicts.
